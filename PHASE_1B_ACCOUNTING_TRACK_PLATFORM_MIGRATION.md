# Phase 1B — Accounting Track: Platform Migration & No-Code Instructor Authoring

**Prerequisites:** `PHASE_1_MVP_FOUNDATION.md` chunks 1–18 (schema, auth, RBAC, R2, course/lesson CRUD, quiz builder, course player, progress tracking) must be done first — this file does not re-decide infrastructure, it builds on it.
**Does not replace:** `FIX_IT_PLAN.md` (the static-site sidebar/CDN/consistency fixes GLM is executing now). That work still ships — the static `claude-made-course` build stays the SCORM-export path for any third-party LMS deployment (Moodle, Docebo, etc.), per its original design goal. This file is the parallel track that moves the *native, Skarion-hosted* version of the same course onto the real platform.
**Goal:** Take the 171-chunk, 15-module Skarion Accounting Track — currently a hardcoded `content.json` → Python build script → single static HTML file, editable only by hand-writing JSON — and turn it into a database-backed course on the platform from `PHASE_1_MVP_FOUNDATION.md`, where an instructor with zero coding knowledge can create, edit, upload files for, reorder, and publish every part of it through a UI. Nothing currently in the course gets thrown away — it gets imported, not rebuilt from scratch.

**Why this is its own file, not just more Phase 1 chunks:** Phase 1's block-type schema (chunk 15) only specs `text` and `video` block types because it was scoped to the OSP course. The accounting course has **14 distinct interactive chunk types** (a journal-entry debit/credit builder, a spreadsheet lab, a drag-and-drop categorizer, a timed drill, a scenario lab, a script builder, flip cards, hotspots, timelines, two quiz variants, plus content/video/exercise) that don't exist anywhere in the Phase 1 plan. This file is the schema extension, the migration script, and the authoring UI for that richer content model — the single biggest reason this course currently *has* to be hand-edited by someone who can write JSON.

---

## Section A — Content model: stop hardcoding chunk types

1. **Block-type registry.** Define a TypeScript discriminated union covering all 14 block types found in `claude-made-course/modules/*/content.json` (`content`, `video`, `flip_cards`, `quiz`, `review_quiz`, `journal_entry_builder`, `exercise`, `scenario_lab`, `spreadsheet_lab`, `drag_and_drop`, `hotspot`, `timed_drill`, `script_builder`, `timeline`, `completion`). Done when: the union type-checks and a hand-written sample of each type compiles.
2. **Zod schema per block type**, mirroring chunk 1's TS types, used for both API validation and to auto-generate builder forms later (Section B). Done when: posting a malformed block (e.g. a `journal_entry_builder` with unbalanced debit/credit test data) to a validation endpoint returns field-level errors, not a generic 500.
3. **Port the block renderers, don't rewrite them.** `claude-made-course/engine/player.js` already has working, tested rendering + interaction logic for all 14 types (quiz scoring, journal-entry debit/credit validation against a solution key, flip-card animation state, drag-and-drop drop-zone matching, timed-drill countdown). Translate each into a React component 1:1 — same validation rules, same completion criteria — rather than redesigning the interaction model. Done when: each component has a preview story rendering a real chunk (pulled from the actual imported content, not a toy example) and produces the same pass/fail/completion result the static site currently does for that exact chunk.
4. **Extend the `lessons.blocks[]` JSONB column usage** (already schema-flexible per Phase 1 chunk 15's own note — "should support future block types without migration") to store `{ block_type, block_data, position }` per entry. Done when: a single lesson can hold a mixed sequence of all 14 block types with no DB migration required to add a 15th type later.
5. **Submission-capable block types get a `submission_schema`.** Not every block produces a gradeable artifact (a `content` block doesn't; an `exercise`'s reflection textarea does). Tag which block types write to the new `submissions` table (Section C) versus which are purely self-graded client-side (`quiz`, `journal_entry_builder`). Done when: the tagging is enforced by chunk 2's Zod schema, not left as a convention someone can forget.

## Section B — Instructor content authoring (no code required)

6. **RBAC scoping for instructors.** Phase 1 chunk 9 already enforces the 5-tier role model (owner/admin/instructor/TA/student) at the middleware level — this chunk makes the *content* boundary explicit: an instructor can create/edit/publish lessons only within courses they're assigned to (a `course_instructors` join table), never another instructor's course, never tenant settings or billing. Done when: an instructor assigned to the Accounting Track gets 403 attempting to edit a course they're not assigned to, and 200 on their own.
7. **Course Builder shell** at `/instructor/courses/[id]/build` — a left rail listing modules, expandable to lessons, expandable to blocks. **Build the sidebar correctly the first time here.** `FIX_IT_PLAN.md` Tier 1 documents exactly what's wrong with the static site's sidebar (expand/collapse conflated with navigation, no type icons, no completion state, hardcoded colors) — this native rebuild must not repeat those mistakes: expanding a module must not navigate away from what you're editing, every block row shows a type icon, and colors come from the platform's existing design tokens (whatever Phase 1's `shadcn/ui` theme already defines), not new hardcoded hex values. Done when: an instructor can browse the full 15-module, 171-chunk tree without ever losing their place, and a design review confirms zero new hardcoded colors were introduced.
8. **No-JSON block editor forms**, one per block type, generated from chunk 2's Zod schemas: a rich-text editor for `content`; add/remove-option UI for `quiz`; a debit/credit row builder with an account-name dropdown for `journal_entry_builder`; a front/back card-pair builder for `flip_cards`; a bucket/item builder for `drag_and_drop`; etc. Done when: an instructor can edit an existing block's copy, quiz answers, or journal-entry solution entirely through form fields — the only path to raw JSON is an explicitly-opt-in "Advanced" toggle, off by default, that a non-technical instructor never needs to touch.
9. **Live preview pane** in the Course Builder, rendering the block being edited with the *exact* React component (chunk 3) students will see — no separate preview-only renderer to drift out of sync. Done when: editing any field updates the preview instantly, no save/reload required.
10. **File upload wired into block editors.** Any block referencing a file (`exercise`, `spreadsheet_lab`) gets a drag-drop upload zone using Phase 1 chunk 12's presigned R2 upload, restricted to the file types this course actually uses (`.xlsx`, `.docx`, `.pdf`, `.csv`), with a max-size guard and a clear error state (not a silent failure) on rejection. Done when: an instructor replaces an exercise's attached Excel file by dragging a new one onto the block, with zero engineering involvement.
11. **Drag-to-reorder at block level**, extending Phase 1 chunk 11's lesson-level reordering one layer deeper. Done when: reordering blocks within a lesson persists and the student-facing chunk rail reflects the new order immediately, with no page reload.
12. **Draft/publish state**, per lesson and per course. Instructors can save work-in-progress invisible to students; a tenant-level setting can require admin review before publish. Done when: an unpublished lesson edit is invisible in the student course view, and publishing timestamps the change.
13. **Version history**, not destructive overwrites — every block save creates a version row with a simple diff/revert UI. Done when: an instructor can see a block's last 10 edits and revert to any of them in one click.
14. **JSON import/export for power users**, using the exact shape from the Section D migration script. This keeps "hand-author in JSON" alive as an optional path (useful for bulk edits, or for another tenant importing this course) without making it the *primary* authoring UX the way it is today. Done when: exporting a module then re-importing it produces an identical lesson tree, byte-for-byte on the content fields.

## Section C — Student experience: real accounts, real submissions, real portfolio

15. **Submission storage**, replacing the current reflection `<textarea>` that's explicitly labeled *"saved locally in this browser only"* in the static build — meaning no instructor has ever been able to see a single student's reflection answer. New `submissions` table: `lesson_id`, `block_id`, `student_id`, `content`, `submitted_at`, `feedback`, `feedback_at`. Done when: a submission survives a browser clear/different-device login (proving it's server-side, not localStorage) and is visible to the assigned instructor.
16. **Instructor grading/feedback UI** for submission-bearing blocks (`exercise` reflections, `scenario_lab`, `script_builder`). A simple approve/comment flow, not a full rubric system yet. Done when: a student sees their instructor's comment attached to their own submission on next login.
17. **MVR (Minimum Viable Readiness) auto-tracking.** The course backbone (`courses/accounting-track/00-course-design/Accounting_Track_V1_Backbone.md`, Part 5) defines 10 specific readiness criteria (e.g. "reconcile a bank statement and explain discrepancies found"). Compute each criterion's status from actual completed blocks — never a manually-ticked checkbox that can be gamed or forgotten. Done when: completing the bank-reconciliation module's required blocks automatically flips MVR item #5, visible on both the student's own dashboard and their instructor's roster view.
18. **Portfolio export.** The backbone specs 8 real work-product artifacts a graduate should be able to show an employer (QuickBooks file, 3 months of financial statements, bank rec reports, AR/AP aging, Excel dashboard, close checklist, CEO memo, journal entry log). Build a "My Portfolio" page compiling a student's completed submissions and SkarionBooks exports (Section E) into one downloadable package. Done when: a student who's finished the course generates their portfolio zip/PDF in one click, with zero manual file-hunting.
19. **Certificate generation** on full course/cohort completion (all required MVR items satisfied). Done when: completing the last required block makes a completion certificate (student name, course, cohort dates) available for download with no manual instructor action.

## Section D — Migration: import everything that already exists, break nothing

20. **Content-import script**, structurally identical in spirit to Phase 1 chunk 19's OSP importer. Reads all 15 `claude-made-course/modules/*/content.json` files and inserts them as one course + 15 modules + 171 lessons/blocks, preserving exact chunk order, every word of copy, every quiz answer key, every journal-entry solution. Done when: the imported course, played back through the new platform, is content-identical to the current static build — verified by an automated diff of rendered text per chunk, not spot-checking by eye.
21. **File asset migration.** Every file currently served via the static build's `/download/...` paths (the real docx/xlsx/pdf/csv files under `courses/accounting-track/`) gets uploaded into R2 under the imported course's tenant-scoped path, and every block's file reference gets rewritten to the new R2 key. Done when: a script asserts zero broken file references post-migration — reusing the exact resolution logic `build_course.py`'s `find_asset()` already proved correct, so nothing that currently resolves stops resolving.
22. **Video script migration.** The 23 `video`-type blocks already have full timestamped transcripts (in `content.json` and mirrored as standalone files under `video-scripts/`). Import the transcript as the block's fallback display text now; leave a `video_url` field null until Veo3-generated video exists. Done when: a `video` block with no video file renders its transcript cleanly (matching current static-site behavior), and adding a real video URL later requires zero schema change.
23. **Vendor list / company-data integrity check.** `courses/accounting-track/shared/MASTER_COMPANY_PROFILE.md` is the single source of truth for Skarion Manufacturing's vendors, customers, and key figures ($336,250.00 adjusted bank balance, the $10,547.10 duplicate deposit, etc.) — `FIX_IT_PLAN.md` Tier 3 item 17 already flagged some drift between modules. Run the consistency check *before* migration, not after, so the imported DB starts clean instead of importing a known inconsistency permanently into a harder-to-bulk-edit system. Done when: every canonical figure appears identically across all imported modules, verified by an automated grep-equivalent over the import payload before it's committed to the DB.
24. **Parallel-run cutover.** Run the new platform-hosted course alongside the existing static SCORM build for one full cohort. Compare completion data; treat any content-fidelity gap as a migration bug, not an acceptable loss. Done when: one full cohort completes entirely on the new platform with zero data-loss incidents, at which point the static build is archived (kept, not deleted — it remains the SCORM-export path for third-party LMS deployment) and the platform version becomes the default.

## Section E — SkarionBooks (the QuickBooks clone)

25. **Embed, don't rewrite.** `skarionbooks/` is a working, self-contained vanilla-JS SPA — keep it that way rather than porting it into React, which would be weeks of work for no student-facing benefit. Embed it in the relevant lesson via iframe. Done when: SkarionBooks loads and functions identically inside the platform's lesson player as it does standalone today.
26. **Completion sync bridge.** Add a small `postMessage` handler so SkarionBooks can report events ("reconciliation complete," "P&L generated") to the parent app, which writes them into the Section C submissions table. This gives instructors visibility into SkarionBooks work without touching SkarionBooks' internals. Done when: completing a task inside the embedded SkarionBooks marks the corresponding platform lesson block complete in the student's progress record.
27. **Build the still-missing reconciliation screen.** Carried forward from `FIX_IT_PLAN.md` Tier 4, item 22 — `skarionbooks/app.js` has no `renderReconcile`, only a hint string. This blocks chunk 26's completion-sync for the entire bank-reconciliation module specifically, so it has to land before that module can be considered migrated. Done when: a student can match/clear transactions inside SkarionBooks and reach a $0.00 difference, matching what the Excel-based lab already teaches.

## Section F — Admin & cohort operations

28. **Cohort model.** This course runs as a 10-day intensive with a specific instructor and a specific group of students — not open self-paced enrollment. Add a `cohorts` table (`course_id`, `instructor_id`, `start_date`, `end_date`) and enroll students into a cohort, not just a course. Done when: an admin creates a cohort, assigns an instructor, and bulk-enrolls a list of students by email in one flow.
29. **Instructor roster dashboard**, per cohort: every student's per-module completion %, live MVR status (chunk 17), and a queue of submissions awaiting feedback. Done when: opening a cohort immediately shows an instructor who's behind and what needs grading, with no direct database access required.
30. **Admin analytics + documentation.** Extend Phase 1 chunk 26's basic analytics with course-specific numbers: average time-to-complete per module, most-missed quiz questions per cohort (surfaced back to instructors as a content-quality signal), and placement-outcome tracking per cohort (per `courses/accounting-track/feedback-for-nuzhat/NEXT_STEPS.md` item 13). Ship alongside two plain-language guides — "How to build a lesson" for instructors and "How to run a cohort" for admins — linked from inside the product, not buried in a repo README. Done when: a non-technical team member follows the instructor guide alone and successfully builds one new module from scratch without asking an engineer for help.

## Suggested execution order
Section A (schema) → Section D chunks 20–23 (migrate existing content early, so every later chunk is built/tested against real data, not toy fixtures) → Section B (authoring UI) → Section C (student experience) → Section E (SkarionBooks) → Section F (admin/cohort) → Section D chunk 24 (cutover, last).

## What this explicitly does not cover
Infrastructure choices (stack, hosting, multi-tenancy model, payments) — those are decided in `GLM_BUILD_PLAN.md` and `PHASE_1_MVP_FOUNDATION.md` and this file doesn't revisit them. AI tutor/RAG features (Phase 1 chunks 23–25) apply to this course once imported with no extra work needed here. Phase 2/3/4 features (gamification, white-label, plugin ecosystem, enterprise compliance) are out of scope for this file by design — this is specifically "get the accounting course, and only the accounting course, onto the real platform with real instructor/student/admin roles."
